# ✨ Aeolus Link Analysis

> 一个专门用来**解析 Aeolus / 风神看板链接**的 OpenClaw Skill。  
> 它不只是“看页面”，而是尽量把一条查询背后的 **Query 身份、字段口径、筛选逻辑、SQL、结果值和风险点** 全部拆开讲明白。

---

## 🎯 这个 Skill 是干什么的？

当你把一条 Aeolus / 风神链接丢给 OpenClaw 时，这个技能会尽量把它还原成：

- **这是谁的查询**：appId、Query ID、Snapshot ID、页面类型
- **这条查询在看什么**：核心维度、核心指标、关键筛选
- **底层怎么跑出来的**：原始 SQL、伪 SQL、expr/fullExpr、派生字段展开
- **业务口径到底是什么**：字段/指标/筛选条件的真实含义
- **有没有坑**：去重风险、时间口径、派生字段歧义、空值处理、过滤遗漏等

一句话说：

> **它把“一个风神链接”翻译成“人能看懂、能复核、能落文档的完整查询说明书”。**

---

## 🚀 核心能力

### 1) Link Fingerprinting（链接指纹识别）
先从 URL 本身提取关键身份信息：

- `appId`
- `id`
- `sid`
- `extra`
- 其他可解码参数

这样能先判断：
- 这是不是查询页 / 看板页 / 快照页
- 输入的 `id` 是不是后续可用的真实 `queryHistoryId`
- 是否存在页面跳转后会变化的查询身份

---

### 2) 页面结构化解析优先
这个 Skill 的默认策略不是傻点 UI，而是优先走**结构化路径**：

- `getReportInfoForllmForVizQuery()`
- `getDataQueryDatasetForAgent()`
- 状态树 / store / schema / component state
- 底层元数据接口
- `/api/v3/misc/sql?queryHistoryId=...&detail=true`

也就是说，它优先把页面当成一个**已登录的运行环境**，直接读取结构化数据，而不是靠肉眼截图猜。

---

### 3) SQL 还原
优先尝试拿：

- `data.sql`
- `data.query_logic.full_sql`
- `sqlList`
- `expr / fullExpr / aeolusFullExpr`

并建立映射：

- 页面字段名 ↔ 字段 ID
- 字段 ID ↔ SQL 列 / `_col_xxx`
- 中文业务名 ↔ 表达式逻辑

这样不仅能拿到 SQL，还能知道 **SQL 里的每一段逻辑在页面上对应什么业务概念**。

---

### 4) 派生字段递归展开
如果页面里出现：

- `[xxx]`
- `_col_xxx`
- alias
- CASE WHEN 中间层字段
- 指标引用指标 / 字段引用字段

这个 Skill 会尽量继续往下展开，而不是停留在一层“翻译了一半”的名字上。

目标是尽量把：

> “页面字段名” → “中间字段” → “最终 SQL 逻辑”

完整串起来。

---

### 5) 飞书文档标准化交付
这套 Skill 默认不是只吐一段聊天回复，而是采用**两段式交付**：

#### 第一段：聊天里给简洁结论
包括：
- 查询身份
- 核心结论
- 风险提醒

#### 第二段：完整解析写入飞书文档
固定输出 10 个章节：

1. 查询身份
2. 一句话结论
3. 维度 SQL 逻辑表
4. 指标 SQL 逻辑表
5. 筛选项 SQL 逻辑表
6. 关键派生字段递归展开
7. 合并后的 WHERE 逻辑
8. 原始 SQL / 伪 SQL
9. 结果值
10. 风险点

而且第 **3 / 4 / 5 / 9** 节默认必须落成表格，不允许随便散着写。

---

## 🧠 为什么这个 Skill 有用？

很多 Aeolus / 风神链接的问题都不是“看不到页面”，而是：

- 看到了结果，但**不知道指标怎么算的**
- 看到了字段名，但**不知道底层 SQL 是什么**
- 看到了 SQL，但**不知道业务口径是什么意思**
- 看到了页面筛选，但**不知道是否还有隐藏过滤条件**
- 看到了 `_col_123`，但**不知道它到底对应哪个字段**

这个 Skill 的价值就在于：

> **把“页面表象”拆回“查询逻辑本体”。**

---

## 🛠 默认执行链路

这是当前沉淀出来的最优路径：

```text
Link Fingerprinting
  → 打开页面
  → getReportInfoForllmForVizQuery()
  → getDataQueryDatasetForAgent()
  → 确认真实 queryHistoryId
  → 调 /api/v3/misc/sql?detail=true
  → 提取 data.sql / full_sql / sqlList
  → 字段映射与递归展开
  → 用 vizData 校验结果值
  → 输出对话总结 + 飞书文档
```

这条链路的核心思想是：

- **先抓身份**
- **再抓结构化信息**
- **再抓 SQL**
- **最后做业务解释与风险扫描**

而不是反过来。

## 📦 适用场景

当你想让 OpenClaw：

- 解析一条 Aeolus / 风神看板链接
- 看懂 Query ID 对应的查询逻辑
- 找出这条查询真实执行的 SQL
- 解释页面维度 / 指标 / 筛选的业务口径
- 把解析结果整理成可复核的飞书文档
- 排查结果值与 SQL 逻辑是否一致
- 找出查询定义里的潜在风险点

就该用这个 Skill。

## 📄 输出风格

这个 Skill 默认追求的是：

- **前面短**：聊天里先给简洁结论
- **后面全**：详细结果写文档
- **结构化**：维度 / 指标 / 筛选 / 结果值优先表格化
- **可复核**：尽量保留原始 SQL，不做过度“美化”
- **有解释**：不仅给技术逻辑，还翻译业务含义
- **有风险意识**：主动指出可能的口径问题

---

如果你需要的是“看懂一个 Aeolus 链接到底在算什么”，它就是干这个的。
